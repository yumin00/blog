## 들어가며

HealthKit/Health Connect 연동 기능을 설계하다가, 처리 흐름을 그려보고 멈췄다. 데이터 하나를 저장하는 데 DB/Redis 호출이 28번이었다.

처음에는 단순하게 생각했다. 앱이 건강 데이터를 보내면, 서버가 받아서 DB에 저장하면 끝이라고. 그런데 실제로는 **검증 → 중복 체크 → 응답 저장 → 데이터 저장 → 상태 업데이트**가 얽혀 있었고, 이게 항목마다 순차적으로 반복되는 구조였다.

이 글에서는 세 가지 레이어에서 겪은 문제와 해결 과정을 정리한다:

1. **서버 처리 최적화** — 순차 I/O를 병렬화하고, 데이터 정합성을 트랜잭션으로 보장
2. **운영에서 터지는 문제** — 에러 로그 없이 데이터가 사라지는 유령 PENDING, 동시 접속 폭증
3. **페이로드 분할 판단** — 300KB를 쪼갤 것인가, 한 번에 보낼 것인가

---

## 먼저, 멱등성(Idempotency)

이 글에서 계속 등장하는 핵심 개념이라 먼저 정리한다. 멱등성은 **같은 요청을 여러 번 보내도 결과가 동일한 것**을 의미한다.

왜 필요한가? 모바일 환경에서는 이런 상황이 흔하다:

```
앱이 데이터 제출 → 네트워크 타임아웃 → 서버는 실제로 저장함
앱: "실패한 것 같으니 재시도하자"
앱이 같은 데이터 다시 제출 → 서버에 중복 저장됨!
```

멱등성이 보장되면:

```
앱이 같은 데이터 다시 제출
→ 서버: "이미 처리했네" → 중복 없이 성공 응답 반환
```

이 시스템에서는 **2단계 멱등성**을 구현했다:

```
L1 (Redis, 빠름):   "이 requestId 처리했어?" → 있으면 즉시 반환
L2 (DB, 확실함):    "이 requestId에 응답 있어?" → 있으면 즉시 반환
```

Redis는 메모리 기반이라 빠르지만 재시작하면 사라질 수 있다. DB는 느리지만 확실하다. 두 계층을 조합해서 **빠르면서도 확실한** 멱등성을 구현한 것이다. 이 구조가 뒤에 나오는 모든 개선과 운영 방어의 전제가 된다.

## 문제 상황

앱이 서버에 건강 데이터를 보내는 구조는 이렇다:

```
앱 → POST /external-health/responses

Body:
{
  responses: [
    { requestId: "req-001", type: "heartRate",     records: [...168개] },
    { requestId: "req-002", type: "sleepAnalysis",  records: [...12개] },
    { requestId: "req-003", type: "stepCount",      records: [...7개] },
    { requestId: "req-004", type: "activeCalories", records: [...7개] },
  ]
}
```

4개 항목이 하나의 POST로 들어온다. 서버는 각 항목을 **순차적으로** 처리한다.

### 항목 하나를 처리하는 데 필요한 작업

서버가 `responses[0]` (심박수 데이터)을 처리하는 과정:

```
① Redis GET  — "이 요청 이미 처리했나?" (멱등성 L1 체크)
② DB SELECT  — "req-001 요청이 존재하나?" (요청 유효성)
③ DB SELECT  — "req-001에 대한 응답이 이미 있나?" (멱등성 L2 체크)
④ DB INSERT  — 응답 메타데이터 저장
⑤ Redis SET  — "처리했다" 캐시 저장
⑥ DB INSERT  — 심박수 레코드 168개 일괄 저장 (createMany)
⑦ DB UPDATE  — req-001 상태를 COLLECTED로 변경
```

**항목 하나당 7번의 네트워크 왕복**(DB 또는 Redis)이 발생한다.

### 왜 느린가: 순차 처리의 함정

4개 항목이 하나씩 차례로 처리된다:

```
항목1: ①②③④⑤⑥⑦ → 항목2: ①②③④⑤⑥⑦ → 항목3: ①②③④⑤⑥⑦ → 항목4: ①②③④⑤⑥⑦
──────────────────────────────────────────────────────────────────────→ 시간
```

**4 × 7 = 28번의 네트워크 왕복이 줄줄이 순차 실행된다.**

각 왕복이 3ms(최선)~20ms(부하 시)라고 하면:

| 상황 | 왕복당 시간 | 총 시간 |
|------|-----------|---------|
| DB 여유로움 | 3ms | 28 × 3 = **84ms** |
| 평균 부하 | 10ms | 28 × 10 = **280ms** |
| 고부하 | 20ms | 28 × 20 = **560ms** |

4개 타입에서는 타임아웃(60초)까지 여유가 있다. 하지만 타입이 7개로 늘어나면 49번 왕복, 고부하 시 거의 1초에 달한다. 당장은 괜찮지만, 확장 가능성을 생각하면 최적화할 여지가 충분했다.

## 서버 처리 최적화

### 개선 1: 검증 단계 병렬화

①②③(Redis 체크 + DB 조회 2회)이 4개 항목에 대해 순차적으로 실행된다. 하지만 각 항목의 검증은 **서로 독립적**이다. req-001의 존재 여부를 확인하는 데 req-002의 결과가 필요하지 않다.

```typescript
// 기존: 순차 실행 — 12번의 왕복이 하나씩 줄줄이
for (const item of responses) {
  const cached = await redis.get(item.requestId);      // ①
  const request = await db.findById(item.requestId);    // ②
  const exists = await db.existsByRequestId(item.requestId); // ③
  // ... 저장
}
```

```typescript
// 개선: 검증을 병렬로, 저장은 순차로
const validations = await Promise.all(
  responses.map(async (item) => {
    const [cached, request, exists] = await Promise.all([
      redis.get(item.requestId),           // ① 동시
      db.findById(item.requestId),         // ② 동시
      db.existsByRequestId(item.requestId), // ③ 동시
    ]);
    return { item, cached, request, exists };
  })
);

// 저장 단계는 순차 유지 — 응답 저장 → 데이터 저장 → 상태 업데이트 간에
// 의존성이 있어서 병렬화할 수 없다
for (const { item, cached, request, exists } of validations) {
  if (cached || exists) continue;
  await saveData(item, request);
}
```

```
기존:  항목1(①②③) → 항목2(①②③) → 항목3(①②③) → 항목4(①②③)
       12번 순차 왕복

개선:  항목1(①②③) ─┐
       항목2(①②③) ─┤ 동시 실행
       항목3(①②③) ─┤
       항목4(①②③) ─┘
       3번 왕복 (가장 느린 것만 기다림)
```

검증 단계 왕복이 12번 → 3번으로 **75% 감소**한다.

### 개선 2: 트랜잭션으로 데이터 정합성 보장

④(응답 저장)과 ⑥(데이터 저장)이 별개의 DB 호출이다. ④가 성공한 뒤 ⑥이 실패하면:

```
④ 응답 메타데이터 저장  ✅ → DB에 커밋됨
⑥ 심박수 레코드 저장    ❌ → 실패!
→ "응답은 있는데 데이터가 없는" 불완전한 상태 (partial state)
```

앱 입장에서는 "제출 완료"로 보이지만, 실제 데이터는 누락된 상태다.

```typescript
// 트랜잭션으로 묶으면 전부 성공 또는 전부 실패
await prisma.$transaction(async (tx) => {
  await tx.dataCollectionResponse.create({ data: responseData });  // ④
  await tx.externalHealthData.createMany({ data: healthRecords }); // ⑥
  await tx.dataCollectionRequest.update({                          // ⑦
    where: { id: requestId },
    data: { status: 'COLLECTED' },
  });
});
// 하나라도 실패하면 전부 롤백 → 깨끗한 상태 유지
```

은행 송금을 생각하면 된다. "A 계좌에서 빼기"와 "B 계좌에 넣기"가 반드시 둘 다 성공하거나 둘 다 실패해야 하는 것처럼.

**트레이드오프**: 트랜잭션은 DB 왕복 수를 줄이는 것이 아니라(여전히 동일한 횟수의 쿼리가 실행된다), **정합성을 보장하는 것**이다. 그리고 트랜잭션은 관련 행에 락(lock)을 잡기 때문에, 같은 `requestId`로 동시 요청이 들어오면 두 번째 요청이 락 대기 상태에 빠질 수 있다. 다만 이 시스템에서는 멱등성 L1(Redis)이 중복 요청을 먼저 걸러내기 때문에, 동일 `requestId`가 트랜잭션까지 도달할 확률은 매우 낮다.

### 개선 3: Redis 선행 필터링

현재도 항목별로 Redis L1 체크를 하고 있지만, 한 단계 더 최적화할 수 있다. 모든 항목의 Redis 체크를 **먼저 한꺼번에** 실행하고, 캐시 미스인 항목만 DB 검증으로 넘기는 것이다.

```typescript
// 1단계: Redis에서 이미 처리된 항목을 한꺼번에 필터링
const cacheResults = await Promise.all(
  responses.map(item => redis.get(`idempotency:${item.requestId}`))
);

const uncachedResponses = responses.filter((_, i) => !cacheResults[i]);

// 전부 캐시 히트면 여기서 끝 — DB 호출 0번
if (uncachedResponses.length === 0) return { results: [...] };

// 2단계: 캐시 미스 항목만 DB 검증 진행
```

**두 번째 접속 시**: 이전에 처리한 항목은 Redis 캐시에 있으므로 DB를 전혀 건드리지 않고 즉시 반환한다. 이 최적화는 아직 적용 전이지만, 중복 제출이 빈번한 환경이라면 적용할 계획이다.

## 구현하면서 미리 잡은 문제들

서버 처리 최적화와 별개로, 구현 과정에서 "이건 운영에서 터지겠다"고 판단한 문제들이 있었다. 코드 리뷰와 시나리오 분석 단계에서 미리 발견하고 대응한 케이스들을 정리해봤다.

### 유령 PENDING — 에러 로그 없이 데이터가 빠지는 문제

이 시스템은 서버가 "이 기간의 데이터를 줘"라는 요청(PENDING)을 만들고, 앱이 HealthKit에서 수집해서 응답하는 Request-Response 패턴이다.

문제 시나리오:

```
1. 유저가 앱을 열음
2. 서버: PENDING 요청 21개 생성 (7일 × 3타입)
3. 앱: HealthKit에서 데이터 수집 시작...
4. 수집 도중 유저가 앱을 닫음 (전화, 배터리 등)
5. 서버에 제출 안 됨 → 21개 전부 PENDING 상태로 남음
```

다음에 접속하면?

```
서버: "PENDING이 21개 있네" → 기존 것 그대로 반환 (새로 안 만듦)
```

이 PENDING들의 기간은 처음 만들었을 때 기준이다. 3일 뒤에 접속해도 서버는 옛날 기간 요청을 반환한다. **그 사이 기간은 아무도 요청하지 않아서 데이터가 빠진다.**

왜 무서운가:

- 서버 에러 로그: **없음** (정상 응답했으니까)
- 앱 에러: **없음** (다음에 정상 수집하니까)
- 유저: **모름**
- 나중에: "왜 이 유저는 3월 데이터가 비어있지?" → 원인 찾기 매우 어려움

**해결**: PENDING에 만료 시간(`expiresAt`)이 이미 DB에 있었다. 문제는 조회 시 이 값을 체크하지 않고 PENDING이면 무조건 반환하고 있었던 것이다. 만료된 PENDING을 EXPIRED로 처리하고, 빠진 기간을 포함해서 새로 생성하는 로직을 추가했다.

**교훈: 에러가 나지 않는 버그가 가장 무섭다.** 상태 머신을 설계할 때 "정상 흐름"만 생각하면 이런 구멍이 생긴다. "중간에 끊기면?"을 항상 질문해야 한다.

### 대량 INSERT 동시 발생 — 월요일 아침 효과

주말에 안 들어온 유저들이 월요일에 한꺼번에 접속하는 상황:

```
유저A: 7일치 심박수 → createMany(168 records)
유저B: 7일치 심박수 → createMany(168 records)
유저C: 7일치 심박수 → createMany(168 records)
... 동시에 100명 = 16,800개 레코드가 동시에 DB INSERT
```

`createMany` 자체는 효율적이다. 하지만 **동시에 100개가 실행될 때** DB 커넥션 풀이 문제가 된다:

```
DB 커넥션 풀: 보통 10~20개

유저1~10의 createMany → 커넥션 10개 점유 (각 50ms)
유저11~100 → 커넥션 없음 → 대기 → 누적 → 타임아웃 위험
```

여기에 unique 제약 체크가 추가된다. `ExternalHealthData`에 `@@unique([userId, type, dayIndex, timestamp])`가 걸려 있고, `skipDuplicates: true`를 사용하면 **매 행마다 인덱스 룩업이 발생**한다. 168행이면 168번 체크. 단순 INSERT보다 2~3배 느려진다.

**대응**: 이 문제는 코드 최적화보다 **인프라 레벨**에서 대응해야 한다. Cloud Run의 auto-scaling으로 컨테이너가 늘어나면 커넥션 풀도 분산된다. 다만 커넥션 풀 총량이 DB의 `max_connections`를 넘지 않도록 관리해야 한다. Cloud SQL Proxy를 사용하면 커넥션 풀링을 인프라 레벨에서 처리해준다.

## 페이로드 크기: 쪼갤 것인가, 한 번에 보낼 것인가

API를 설계할 때, 클라이언트가 보내는 페이로드 크기에 대한 판단이 필요했다. "큰 데이터는 무조건 쪼개야 한다"는 말은 맞지만, **어디서부터가 "크다"인지**가 중요하다.

### 우리 데이터는 얼마나 큰가

```
Apple Watch 심박수 (5분 간격, 7일치):
  60/5 × 24 × 7 = 2,016 레코드 × ~110바이트 = ~220KB

수면 + 걸음 + 칼로리 포함:
  총 ~300KB 하나의 POST body
```

### 실제 서비스들은 어떻게 하는가

**Strava** — GPS 데이터, 수천~수만 포인트:
- 단일 multipart POST (최대 25MB)
- 쪼개지 않는다. 대신 서버가 즉시 "접수" 응답 후 백그라운드 처리
- 하루 수십억 GPS 포인트 처리 규모

**Fitbit** — 심박수, 수면, 걸음수:
- 데이터 타입별로 분리된 REST API (`/heart-rate`, `/sleep`, `/steps`)
- 하지만 이건 네트워크 최적화가 아니라 API 설계 차원의 분리

**Google Health Connect** — 건강 데이터 동기화:
- 변경 토큰(Change Token) 기반 델타 동기화
- 첫 동기화만 전체 배치, 이후는 변경분만 전송

### 업계 기준: 분할이 필요한 크기

| 크기 | 업계 판단 | 출처 |
|------|----------|------|
| ~1MB | 단일 전송 OK | 대부분의 서비스 |
| 1~20MB | 상황에 따라 | Strava (25MB까지 단일) |
| 20MB+ | 분할 필수 | Box API (20MB부터 청크 적용) |

우리 데이터(300KB)는 분할이 필요한 크기에 한참 못 미친다.

### 쪼개면 오히려 손해인 경우

300KB를 4개로 나눠 보내면:

```
단일 전송 (300KB × 1회):
  TCP 핸드셰이크 1번 + 전송 0.3초 + 서버 처리 1번 = ~0.5초

분할 전송 (75KB × 4회):
  TCP 핸드셰이크 4번 + 전송 0.1초 × 4 + 서버 처리 4번
  + 매 요청마다 JWT 인증 검증 = ~0.8초
```

전송 시간이 오히려 길어지고, 서버 부하도 4배가 된다. **전송 시간이 길수록 그 사이에 네트워크가 끊길 확률도 높아진다.**

### 서버 관점에서 진짜 중요한 것

클라이언트가 300KB를 한 번에 보내든 나눠 보내든, 서버 입장에서 더 중요한 건:

1. **멱등성**: 같은 데이터가 두 번 와도 중복 저장 안 되는가
2. **재시도 안전성**: 클라이언트가 실패 후 통째로 재전송해도 문제없는가
3. **타임아웃**: 300KB 수신 + 처리가 서버 타임아웃(60초) 안에 끝나는가

이 세 가지가 보장되면, 300KB를 한 번에 받아도 안전하다. 실제로 현재 시스템은 Redis + DB 이중 멱등성 체크로 1번과 2번을 보장하고 있고, 처리 시간도 최악의 경우 560ms로 타임아웃과는 거리가 멀다.

### 그래도 클라이언트 환경은 고려해야 한다

백엔드 개발자가 API를 설계할 때 모바일 환경의 특성을 완전히 무시할 수는 없다. 모바일 클라이언트 개발자와 협의하면서 고려한 점들:

- **iOS**: `NSURLSession` 백그라운드 세션을 사용하면 앱이 닫혀도 OS가 전송을 완료해준다. iOS 17부터는 중단 지점부터 재개(Resumable Upload)도 가능하다.
- **Android**: `WorkManager`로 전송을 예약하면 앱 종료/재부팅 후에도 지수 백오프로 자동 재시도한다.

**결론: "데이터를 쪼갤까 말까"보다 "앱이 닫혀도 전송이 완료되는가"가 모바일 환경에서 더 중요한 문제**였다. 서버는 멱등성만 잘 보장하면, 클라이언트가 어떤 방식으로 재시도하든 안전하다.

## 정리

### 개선 전후 비교 (4개 항목 기준)

| 지표 | 현재 | 개선 후 |
|------|------|---------|
| 검증 DB 왕복 | 12번 (순차) | 3번 (병렬) |
| 저장 DB 왕복 | 16번 (순차) | 16번 (트랜잭션 묶음) |
| 중복 요청 시 | DB 4회 추가 | Redis 4회로 즉시 반환 |
| partial state 위험 | 있음 | 없음 (트랜잭션) |
| 평균 처리 시간 | ~280ms | ~190ms |

> 개선 전: 검증 12번 + 저장 16번 = 28번 × 10ms = 280ms
> 개선 후: 검증 3번(병렬) + 저장 16번 = 19번 × 10ms = 190ms
> 제목의 "12번 → 3번"은 검증 단계 기준 — 저장 단계는 의존성이 있어 순차 유지

## 마치며

이번 작업을 통해 가장 크게 느낀 건, **"각 작업이 서로 의존하는가?"를 먼저 분석하는 것**이 서버 최적화의 출발점이라는 것이다. 의존성이 없으면 병렬화할 수 있고, 의존성이 있으면 트랜잭션으로 묶어야 한다. 그리고 코드 밖의 문제 — 앱 종료로 인한 상태 구멍, 동시 접속 폭증, 페이로드 분할 판단 — 는 구현 단계에서 시나리오를 그려보면 미리 잡을 수 있다.

특히 **에러가 나지 않는 문제가 가장 무섭다**는 걸 다시 한번 느꼈다. 유령 PENDING 같은 케이스는 모니터링에도 안 잡히고, 서버 로그에도 안 남는다. 상태 머신을 설계할 때 "정상 흐름"만 생각하면 안 되고, **"중간에 끊기면?"을 항상 질문**해야 한다.
